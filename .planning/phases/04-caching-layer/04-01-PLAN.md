    **Step 2: Add ///// Caching Layer ///// section**

    Insert between the `///// Response Parsers /////` section (ends at line 1127) and the `///// CLI Interface /////` placeholder (line 1129). Insert the following four functions:

    ```javascript
    ///// Caching Layer /////

    /**
     * Get cache file path for a given provider.
     * Per D-03: Uses os.tmpdir() for cross-platform temp directory.
     * Per D-04: Filename pattern cc-usage-${provider}-cache.json.
     * @param {string} provider - Provider name ('kimi' or 'glm')
     * @returns {string} Absolute path to cache file
     */
    function getCacheFilePath(provider) {
      return join(tmpdir(), `cc-usage-${provider}-cache.json`);
    }

    /**
     * Read cache file and check TTL expiration.
     * Per D-05: Strict TTL -- expired cache returns null.
     * Per D-08: Fail-open -- any error returns null (never throws).
     * @param {string} filePath - Path to cache file
     * @param {number} maxAgeMs - Maximum age in milliseconds
     * @returns {Promise<object|null>} Cached data or null if expired/missing
     */
    async function readCache(filePath, maxAgeMs) {
      try {
        const content = await readFile(filePath, 'utf-8');
        const data = JSON.parse(content);

        // Strict TTL check per D-05
        if (Date.now() - data.timestamp > maxAgeMs) {
          return null;
        }

        return data;
      } catch {
        return null; // Fail-open per D-08
      }
    }

    /**
     * Write data to cache file atomically using write-then-rename.
     * Per D-07: Write to temp file with process.pid suffix, then rename.
     * Per D-08: Fail-open -- errors are logged to stderr but not thrown.
     * @param {string} filePath - Target cache file path
     * @param {object} data - Data to cache
     * @returns {Promise<void>}
     */
    async function writeCache(filePath, data) {
      const tempPath = `${filePath}.${process.pid}.tmp`;
      try {
        await writeFile(tempPath, JSON.stringify(data));
        await rename(tempPath, filePath);
      } catch (error) {
        // Clean up temp file on failure
        try { await unlink(tempPath); } catch {}
        // Fail-open per D-08: don't throw, log to stderr silently
        // Per D-09: No debug output. Error is swallowed.
      }
    }

    /**
     * Get usage data with cache-first strategy.
     * Per D-06: Blocking refresh -- wait for API when cache miss.
     * Per D-10: Cache-first flow: read cache -> return if valid -> API call -> write cache -> return data.
     * Per D-02: Cache structure { timestamp, provider, total, used, remaining, percent, reset_display }.
     * @param {number} [cacheDuration=DEFAULT_CONFIG.cacheDuration] - Cache TTL in seconds
     * @returns {Promise<{ total: number, used: number, remaining: number, percent: number, reset_display: string, provider: string }>}
     */
    async function getCachedUsageData(cacheDuration = DEFAULT_CONFIG.cacheDuration) {
      // Step 1: Fetch fresh data from API (getUsageData detects provider internally)
      const data = await getUsageData();

      // Step 2: Determine cache file path based on provider from response
      const filePath = getCacheFilePath(data.provider);
      const maxAgeMs = cacheDuration * 1000;

      // Step 3: Try cache read first
      const cached = await readCache(filePath, maxAgeMs);
      if (cached) return cached;

      // Step 4: Cache miss -- write new data to cache (fire-and-forget per D-10)
      writeCache(filePath, {
        timestamp: Date.now(),
        provider: data.provider,
        total: data.total,
        used: data.used,
        remaining: data.remaining,
        percent: data.percent,
        reset_display: data.reset_display
      }); // Fire-and-forget -- no await per D-10

      // Step 5: Return fresh data
      return data;
    }
    ```

    **IMPORTANT**: The `getCachedUsageData` function flow is slightly different from the CONTEXT.md example. If cache is valid, return cached data. If not, return the fresh data we already fetched and write it to cache.

If cache miss/expired -> call getUsageData() -> write new data to cache -> return new data." This means cache read should happen BEFORE API call.     **Resolution per Open Question 1 in RESEARCH.md**: Use approach (b) -- use a generic cache filename that stores the provider inside the cache data.     **Correct approach**: Since D-04 explicitly says `cc-usage-${provider}-cache.json` and Phase 2 already detects the provider (via CC Switch credentials), the provider is known BEFORE calling `getUsageData()`. The `getCredentials()` function returns `{ provider }`. So `getCachedUsageData` should accept an optional `provider` parameter, or we should resolve credentials first.

    **Simplest correct approach per the codebase**: The tool is designed for CC Switch proxy where provider is always known upfront (detected from base URL). For env-only usage without CC Switch, `getCredentials()` returns `provider: null`. But in practice, CC Switch is the primary use case.

    **Final implementation**: Make `getCachedUsageData` accept an optional `provider` parameter. When provider is known (from CC Switch), check cache before API call. When provider is unknown, skip cache check and call API directly, then write to cache using the provider from the response.
